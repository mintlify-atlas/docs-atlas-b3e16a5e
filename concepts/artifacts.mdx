title: "Artifacts"
description: "Artifacts are the documents within a change that guide the work"
---

Artifacts are the documents that live inside a change folder. Each artifact has a specific role in the workflow. Together they capture why a change is being made, what it will do, how it will be built, and what steps to take.

## The four artifact types

| Artifact | File | Role |
|---|---|---|
| Proposal | `proposal.md` | Why and what — intent, scope, approach |
| Specs | `specs/**/*.md` | What changes — delta specs for modified behavior |
| Design | `design.md` | How — technical approach and architecture decisions |
| Tasks | `tasks.md` | Steps — numbered implementation checklist |

## The artifact flow

```text
proposal ──────► specs ──────► design ──────► tasks ──────► implement
    │               │             │              │
   why            what           how          steps
 + scope        changes       approach      to take
```

Artifacts build on each other. Each artifact provides context for the next. The dependency graph enforced by the schema ensures that downstream artifacts have the information they need.

## Each artifact in detail

### Proposal (`proposal.md`)

The proposal captures **intent**, **scope**, and **approach** at a high level. It is the root of the dependency graph — specs, design, and tasks all depend on it.

```markdown
# Proposal: Add Dark Mode

## Why
Users have requested a dark mode option to reduce eye strain
during nighttime usage and match system preferences.

## What Changes
- Theme toggle in user settings
- System preference detection on first load
- Preference persisted in localStorage

## Capabilities

### New Capabilities
- `ui-theming`: Theme switching with light/dark modes and system detection

### Modified Capabilities
(none)

## Impact
- `src/contexts/ThemeContext.tsx` (new)
- `src/components/ThemeToggle.tsx` (new)
- `src/styles/globals.css` (modified)
```

**When to update the proposal:**
- Scope changes (narrowing or expanding)
- Intent clarifies (better understanding of the problem)
- Approach fundamentally shifts

### Specs (delta specs in `specs/`)

Delta specs describe **what's changing** relative to the current specs. Each file covers one domain and uses ADDED / MODIFIED / REMOVED sections to describe exactly what requirements are changing.

```markdown
# Delta for UI

## ADDED Requirements

### Requirement: Theme Selection
The system SHALL allow users to select between light and dark themes.

#### Scenario: Manual theme toggle
- GIVEN an authenticated user
- WHEN the user toggles the theme in settings
- THEN the selected theme is applied immediately
- AND the preference is persisted across sessions

#### Scenario: System preference detection
- GIVEN a user with no saved theme preference
- WHEN the application loads
- THEN the theme matches the user's OS color scheme preference
```

See [Changes](/concepts/changes) for full documentation on the delta spec format.

### Design (`design.md`)

The design captures **technical approach** and **architecture decisions**. It explains how the implementation will work, not just what it will do.

````markdown
# Design: Add Dark Mode

## Technical Approach
Theme state managed via React Context to avoid prop drilling.
CSS custom properties enable runtime switching without class toggling.

## Architecture Decisions

### Decision: Context over Redux
Using React Context for theme state because:
- Simple binary state (light/dark)
- No complex state transitions
- Avoids adding Redux dependency

### Decision: CSS Custom Properties
Using CSS variables instead of CSS-in-JS because:
- Works with existing stylesheet
- No runtime overhead
- Browser-native solution

## Data Flow
```
ThemeProvider (context)
       │
       ▼
ThemeToggle ◄──► localStorage
       │
       ▼
CSS Variables (applied to :root)
```

## File Changes
- `src/contexts/ThemeContext.tsx` (new)
- `src/components/ThemeToggle.tsx` (new)
- `src/styles/globals.css` (modified)
````

**When to update the design:**
- Implementation reveals the approach won't work
- A better solution is discovered
- Dependencies or constraints change

<Note>
  Design is optional. Only create `design.md` when there's genuine architectural complexity: cross-cutting changes, new external dependencies, security or migration concerns, or ambiguity that benefits from documented decisions before coding starts.
</Note>

### Tasks (`tasks.md`)

Tasks are the **implementation checklist** — concrete steps with checkboxes that can be tracked and checked off as work progresses.

```markdown
# Tasks

## 1. Theme Infrastructure
- [ ] 1.1 Create ThemeContext with light/dark state
- [ ] 1.2 Add CSS custom properties for colors
- [ ] 1.3 Implement localStorage persistence
- [ ] 1.4 Add system preference detection

## 2. UI Components
- [ ] 2.1 Create ThemeToggle component
- [ ] 2.2 Add toggle to settings page
- [ ] 2.3 Update Header to include quick toggle

## 3. Styling
- [ ] 3.1 Define dark theme color palette
- [ ] 3.2 Update components to use CSS variables
- [ ] 3.3 Test contrast ratios for accessibility
```

**Task best practices:**
- Group related tasks under numbered headings
- Use hierarchical numbering (1.1, 1.2, etc.) so tasks are easy to reference
- Keep tasks small enough to complete in one session
- Use `- [ ]` checkbox format — the apply phase parses this to track progress
- Order tasks by dependency (what must be done first?)
- Check tasks off as you complete them

<Warning>
  Tasks not using the `- [ ] X.Y description` format won't be tracked by the apply phase. Follow the format exactly.
</Warning>

## When to update each artifact

<AccordionGroup>
  <Accordion title="Proposal">
    Update when scope changes (narrowing or expanding), when intent clarifies with a better understanding of the problem, or when the overall approach fundamentally shifts. The proposal is the root of all other artifacts, so changes here may cascade.
  </Accordion>
  <Accordion title="Specs">
    Update when discovered behavior differs from what was originally specified, or when implementation reveals edge cases that weren't captured in the initial scenarios. Specs should reflect what the system will actually do, not what was initially assumed.
  </Accordion>
  <Accordion title="Design">
    Update when implementation reveals the planned approach won't work, when a better solution is discovered during development, or when dependencies or constraints change. The design should always reflect the actual technical direction.
  </Accordion>
  <Accordion title="Tasks">
    Update as implementation progresses — check off completed tasks, add newly discovered tasks, and remove or revise tasks that turn out to be unnecessary. Tasks are a living checklist, not a fixed plan.
  </Accordion>
</AccordionGroup>
