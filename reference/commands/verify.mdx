title: "/opsx:verify"
description: "Validate that implementation matches your change artifacts across three quality dimensions"
---

`/opsx:verify` checks your implementation against the specs, tasks, and design artifacts. It reports issues categorized as CRITICAL, WARNING, or SUGGESTION without blocking the archive workflow.

## Syntax

```text
/opsx:verify [change-name]
```

## Profile

<Warning>
  This is an **expanded workflow** command. To enable it, run `openspec config profile`, select workflows, then run `openspec update` in your project.
</Warning>

## Arguments

<ParamField path="change-name" type="string">
  Which change to verify. If omitted, the AI infers it from context or prompts you to specify one.
</ParamField>

## What it does

- Checks three dimensions of implementation quality
- Searches the codebase for implementation evidence
- Reports issues categorized as CRITICAL, WARNING, or SUGGESTION
- Does not block archive — surfaces issues for you to address or accept

## Verification dimensions

| Dimension | What it validates |
|-----------|-------------------|
| **Completeness** | All tasks done, all requirements implemented, scenarios covered |
| **Correctness** | Implementation matches spec intent, edge cases handled |
| **Coherence** | Design decisions reflected in code, patterns consistent |

## Example

```text
You: /opsx:verify

AI:  Verifying add-dark-mode...

     COMPLETENESS
     ✓ All 8 tasks in tasks.md are checked
     ✓ All requirements in specs have corresponding code
     ⚠ Scenario "System preference detection" has no test coverage

     CORRECTNESS
     ✓ Implementation matches spec intent
     ✓ Edge cases from scenarios are handled
     ✓ Error states match spec definitions

     COHERENCE
     ✓ Design decisions reflected in code structure
     ✓ Naming conventions consistent with design.md
     ⚠ Design mentions "CSS variables" but implementation uses Tailwind classes

     SUMMARY
     ─────────────────────────────────
     Critical issues: 0
     Warnings: 2
     Ready to archive: Yes (with warnings)

     Recommendations:
     1. Add test for system preference detection
     2. Update design.md to reflect Tailwind usage, or refactor to use CSS variables
```

## Tips

<Tip>
  Run `/opsx:verify` before archiving to surface mismatches early. Warnings don't block the archive — you decide whether to address them.
</Tip>

- Good for reviewing AI-generated implementation before committing
- Can reveal drift between artifacts and the final implementation
- If correctness issues appear, fix the code and run verify again before archiving
- Warnings about coherence often indicate the design doc needs updating to reflect implementation decisions
