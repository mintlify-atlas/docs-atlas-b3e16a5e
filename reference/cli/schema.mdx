---
title: "openspec schema"
description: "Create and manage custom workflow schemas"
---

Manages workflow schemas — the artifact graphs that define how a change progresses through proposal, specs, design, and tasks.

<Note>
  Schema commands are experimental and may change in future releases.
</Note>

## Subcommands

| Subcommand | Description |
|------------|-------------|
| `schema init <name>` | Create a new project-local schema |
| `schema fork <source> [name]` | Copy an existing schema for customization |
| `schema validate [name]` | Validate a schema's structure and templates |
| `schema which [name]` | Show where a schema resolves from |

---

## schema init

Creates a new project-local schema with template files.

```bash
openspec schema init <name> [options]
```

### Arguments

<ParamField path="name" type="string" required>
  Schema name. Must be kebab-case (e.g., `my-workflow`).
</ParamField>

### Options

<ParamField path="--description" type="string">
  Short description of the schema.
</ParamField>

<ParamField path="--artifacts" type="string">
  Comma-separated list of artifact IDs to include. Defaults to `proposal,specs,design,tasks`.

  Valid artifact IDs: `proposal`, `specs`, `design`, `tasks`.
</ParamField>

<ParamField path="--default" type="boolean">
  Set the new schema as the project default in `openspec/config.yaml`.
</ParamField>

<ParamField path="--no-default" type="boolean">
  Do not prompt to set as default.
</ParamField>

<ParamField path="--force" type="boolean">
  Overwrite an existing schema with the same name.
</ParamField>

<ParamField path="--json" type="boolean">
  Output result as JSON.
</ParamField>

### What it creates

```
openspec/schemas/<name>/
├── schema.yaml           # Schema definition
└── templates/
    ├── proposal.md       # Template for each artifact
    ├── specs.md
    ├── design.md
    └── tasks.md
```

### Examples

```bash
# Interactive schema creation
openspec schema init research-first

# Non-interactive with specific artifacts
openspec schema init rapid \
  --description "Rapid iteration workflow" \
  --artifacts "proposal,tasks" \
  --default
```

## schema fork

Copies an existing schema into your project directory so you can customize it.

```bash
openspec schema fork <source> [name] [options]
```

### Arguments

<ParamField path="source" type="string" required>
  Name of the schema to copy.
</ParamField>

<ParamField path="name" type="string">
  Name for the new schema. Defaults to `<source>-custom`.
</ParamField>

### Options

<ParamField path="--force" type="boolean">
  Overwrite the destination if it already exists.
</ParamField>

<ParamField path="--json" type="boolean">
  Output result as JSON.
</ParamField>

### Example

```bash
# Fork the built-in spec-driven schema
openspec schema fork spec-driven my-workflow
```

## schema validate

Validates a schema's `schema.yaml` and template files for structural correctness.

```bash
openspec schema validate [name] [options]
```

### Arguments

<ParamField path="name" type="string">
  Schema name to validate. Validates all project schemas if omitted.
</ParamField>

### Options

<ParamField path="--verbose" type="boolean">
  Show detailed validation steps.
</ParamField>

<ParamField path="--json" type="boolean">
  Output result as JSON.
</ParamField>

### Examples

```bash
# Validate a specific schema
openspec schema validate my-workflow

# Validate all project schemas
openspec schema validate

# Verbose output
openspec schema validate my-workflow --verbose
```

**Verbose output example:**

```
Validating my-workflow...
  Checking schema.yaml exists...
  Parsing YAML...
  Validating schema structure...
  Checking template files...
  Dependency graph validation passed (via parseSchema)
✓ Schema 'my-workflow' is valid
```

## schema which

Shows where a schema resolves from, which is useful for debugging schema precedence.

```bash
openspec schema which [name] [options]
```

### Arguments

<ParamField path="name" type="string">
  Schema name to look up.
</ParamField>

### Options

<ParamField path="--all" type="boolean">
  List all schemas with their resolution sources.
</ParamField>

<ParamField path="--json" type="boolean">
  Output as JSON.
</ParamField>

### Example

```bash
# Check where a schema resolves from
openspec schema which spec-driven
```

**Output:**

```
Schema: spec-driven
Source: package
Path: /usr/local/lib/node_modules/@fission-ai/openspec/schemas/spec-driven
```

## Schema precedence

When multiple schemas share the same name, OpenSpec uses the first match in this order:

| Priority | Location | Path |
|----------|----------|------|
| 1 (highest) | Project | `openspec/schemas/<name>/` |
| 2 | User | `~/.local/share/openspec/schemas/<name>/` |
| 3 (lowest) | Package | Built-in schemas |

A project schema with the same name as a built-in schema will shadow the built-in. Use `openspec schema which --all` to see which source each schema resolves from.
